## 2. 安全漏洞清单

### 2.1 高危漏洞

| 编号 | 问题 | 位置 | 说明 |
|------|------|------|------|
| SEC-001 | **缺少登录页面和认证流程** | `frontend/src/router/index.ts` | 路由表中没有 `/login` 路由，所有页面无需登录即可访问。虽然后端 API 已实现 JWT 认证，但前端没有登录界面，也没有路由守卫验证 Token。 |

### 2.2 中危漏洞

| 编号 | 问题 | 位置 | 说明 |
|------|------|------|------|
| SEC-002 | **前端无路由守卫** | `frontend/src/router/index.ts` | `router.beforeEach` 仅设置页面标题，未检查用户是否已登录。未认证用户可以直接访问所有页面。 |
| SEC-003 | **Token 存储策略不明** | `frontend/src/stores/` | 未发现专门的 auth store，Token 的存储（localStorage/sessionStorage）和刷新机制未实现。 |
| SEC-004 | **Token 黑名单仅存内存** | `src/core/security.py:145` | `self._blacklist: set[str] = set()` — 服务重启后黑名单丢失，已吊销的 Token 将重新有效。 |
| SEC-005 | **批量操作无数量限制** | `src/services/grade_service.py:100` | `batch_create_grades` 方法未限制批量大小，可能被用于资源耗尽攻击。 |

### 2.3 低危漏洞

| 编号 | 问题 | 位置 | 说明 |
|------|------|------|------|
| SEC-006 | **CSV 导出未净化** | `frontend/src/views/student/StudentList.vue:413` | CSV 导出时未对数据进行净化处理，存在 CSV 注入风险（低概率）。 |
| SEC-007 | **依赖不一致** | `requirements.txt` | 代码中 `import jwt` 使用 PyJWT 包，但 requirements.txt 列出的是 `python-jose`。实际运行需要 PyJWT 包。 |

### 2.4 已通过的安全检查项

| 检查项 | 状态 | 说明 |
|--------|------|------|
| JWT 认证实现 | ✅ 通过 | HS256 算法、Token 黑名单、过期机制完善 |
| RBAC 权限控制 | ✅ 通过 | admin/teacher/student 三级角色，路由保护完整 |
| CORS 配置 | ✅ 通过 | 已限制来源和方法 |
| SQL 注入防护 | ✅ 通过 | 全 ORM 架构，无原生 SQL |
| XSS 防护 | ✅ 通过 | Vue 3 默认转义，无 v-html 使用 |
| 安全响应头 | ✅ 通过 | 完整的安全头配置 |
| 密码哈希 | ✅ 通过 | bcrypt 哈希存储 |
| 输入验证 | ✅ 通过 | Pydantic + 前端双重验证 |

## 3. 代码质量问题

### 3.1 严重问题

| 编号 | 问题 | 位置 | 说明 | 建议 |
|------|------|------|------|------|
| CODE-001 | **搜索学生性能问题** | `src/services/student_service.py:200` | `search_students` 方法先获取所有匹配记录（limit=10000）到内存，再在 Python 中进行分页。数据量大时会造成严重性能问题。 | 改为数据库层面的分页查询 |
| CODE-002 | **班级列表获取低效** | `frontend/src/stores/student.ts:165` | `fetchClassList` 获取 1000 条学生记录只为提取班级名称列表，极度低效。 | 后端增加专门的班级列表接口 |

### 3.2 一般问题

| 编号 | 问题 | 位置 | 说明 |
|------|------|------|------|
| CODE-003 | **Dashboard 使用硬编码数据** | `frontend/src/views/dashboard/Dashboard.vue:125` | 统计数据为模拟数据（128、1024、78.5、85.2%），未调用真实 API |
| CODE-004 | **AppHeader 用户信息硬编码** | `frontend/src/components/layout/AppHeader.vue:31` | 用户名显示为固定的"管理员"，未从 Token/API 获取真实用户信息 |
| CODE-005 | **退出登录功能未实现** | `frontend/src/components/layout/AppHeader.vue:40` | 退出登录菜单项没有绑定点击事件，点击无反应 |
| CODE-006 | **前端 API 函数名不匹配** | 多处 | 部分 API 调用使用了可能不存在的函数名（如 `checkGradeDuplicate`、`getGradeList`） |

### 3.3 建议改进

| 编号 | 问题 | 位置 | 说明 |
|------|------|------|------|
| CODE-007 | **缺少 API 错误统一处理** | `frontend/src/utils/request.ts` | 建议在 Axios 拦截器中统一处理 401（跳转登录）、403（权限不足）等状态码 |
| CODE-008 | **缺少请求重试机制** | `frontend/src/utils/request.ts` | Token 过期时应自动尝试刷新 Token 并重试请求 |

## 7. 总结

### 项目整体评估

| 维度 | 评分 | 说明 |
|------|------|------|
| 后端架构 | ⭐⭐⭐⭐☆ | 分层清晰，Repository-Service-Router 模式规范 |
| 后端安全性 | ⭐⭐⭐⭐☆ | JWT + RBAC 完善，缺少速率限制 |
| 前端架构 | ⭐⭐⭐⭐☆ | Vue 3 + Pinia + TypeScript 规范 |
| 前端安全性 | ⭐⭐☆☆☆ | 缺少登录页面和路由守卫 |
| 用户体验 | ⭐⭐⭐☆☆ | 基本功能可用，但缺少认证流程和真实数据 |
| 代码质量 | ⭐⭐⭐⭐☆ | 整体规范，个别性能问题 |

### 关键发现

1. **最大的问题**：前端缺少完整的认证流程（登录页面、Token 管理、路由守卫），导致系统无法正常使用
2. **第二大问题**：Dashboard 和 AppHeader 使用硬编码数据，影响用户体验
3. **安全方面**：后端安全措施完善（85/100），但前端认证流程缺失拉低了整体安全性
4. **UI 方面**：当前使用 Element Plus 默认风格，需要美化以符合教育系统温馨友好的定位
